## 4. Correctness requirements

These apply during migration and must be verified before the repo ships.

### Unit conversions

All unit conversions must use `slr_forecast.units` — no hardcoded `* 1000`, `/ 1000`, `* 362.5`, `* 100`.

- Audit: `grep -rn '1000\.\|362\.5\|3\.625e14' slr_forecast/ notebooks/`
- Remove duplicate `M_TO_MM` definitions from component_plotting.py and anywhere else
- Add `assert_units_meters()` at key data-loading boundaries
- Goal: zero hardcoded conversion factors outside `units.py`

### Output precision

Remove all `round()` calls on values written to JSON, HDF5, or other saved files. Save at full float precision. Round only at the display layer (print statements, manuscript text).

- Audit: `grep -rn 'round(' slr_forecast/ notebooks/`

### Comments and documentation

All inline comments and notebook markdown cells must accurately describe the code they accompany. During development, code changes outpaced comments — stale comments are worse than no comments because they mislead.

- Audit every notebook markdown cell: does the section description match what the code actually does?
- Audit inline comments: remove comments that restate the obvious, fix comments that describe behavior the code no longer has, add comments where the logic is not self-evident
- Update variable names that reference the old baseline (e.g., `sq_at_2005` → `sq_at_baseline` — already done in component_forecast.ipynb, check for others)
- Verify docstrings on all public functions in .py modules match their current signatures and behavior

### Numerical equivalence

After migration, run all notebooks end-to-end and compare outputs against the archive repo. Any numerical difference must be explained.
